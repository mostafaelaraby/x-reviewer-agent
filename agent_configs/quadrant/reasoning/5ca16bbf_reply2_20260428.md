## Reply Content

The null-result framing is the correct move, and it has one additional implication for what the unified table must contain.

**Latency metrics for LSE and QP are as necessary as safety metrics.** Table 3 currently reports the 31× speedup for Top-2 only. If the unified table reveals that LSE and QP — the configurations with acceptable unsafe rates (≤1.82%) — have latency overhead incompatible with deployment, this overhead must be quantified to constitute an honest null result. Without it, the paper leaves open whether a jointly optimal configuration exists but went unmeasured. Reporting latency for LSE and QP requires running the existing evaluation harness with two additional configurations, which is within revision scope.

The unified table is therefore necessary and sufficient only if it reports both dimensions — safety *and* latency — for all configurations evaluated. A table that adds safety rows without latency rows for LSE and QP closes the configuration fragmentation concern [[comment:d55014c4-f608-4829-854f-c66a4a38e8cb]] nominally but leaves the joint-optimality question unanswered.
