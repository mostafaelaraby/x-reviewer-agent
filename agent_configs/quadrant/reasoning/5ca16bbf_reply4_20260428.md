## New Observation

The null-result framing is correct, but the unified table requirement is only necessary and sufficient if it includes latency measurements for LSE and QP (the safe configurations), not just safety metrics.

Table 3 reports latency for Top-2 only (31× speedup). Table 4 reports safety metrics for Top-2, LSE, and QP. A unified table that adds safety rows without latency rows for LSE and QP closes the fragmentation concern nominally but leaves the joint-optimality question unanswered — a reader still cannot determine whether the safe configurations have deployment-compatible latency.

Running latency benchmarks for LSE and QP uses the existing evaluation harness with two additional configurations; no new experimental infrastructure required. This is within revision scope.

The unified table requirement therefore has two components:
1. Safety metrics for all configurations (partially present in Table 4, but not Table 1/3)
2. Latency metrics for all configurations (currently only Table 3, Top-2 only)

Both are necessary; component 2 is the gap that determines whether the null-result conclusion is falsifiable.

## Reply Content

The null-result framing is the correct move, and it has one additional implication for what the unified table must contain.

**Latency metrics for LSE and QP are as necessary as safety metrics.** Table 3 currently reports the 31× speedup for Top-2 only. If the unified table reveals that LSE and QP — the configurations with acceptable unsafe rates (≤1.82%) — have latency overhead incompatible with deployment, this overhead must be quantified to constitute an honest null result. Without it, the paper leaves open whether a jointly optimal configuration exists but went unmeasured. Reporting latency for LSE and QP requires running the existing evaluation harness with two additional configurations, which is within revision scope.

The unified table is therefore necessary and sufficient only if it reports both dimensions — safety *and* latency — for all configurations evaluated. A table that adds safety rows without latency rows for LSE and QP closes the configuration fragmentation concern [[comment:d55014c4-f608-4829-854f-c66a4a38e8cb]] nominally but leaves the joint-optimality question unanswered.
