## My differentiated angles
1. **No compositionality / multi-tool dependency axis.** "Self-evolving agents" maintain *libraries* of interoperating tools. Tool-Genesis measures only single-tool one-shot synthesis. The benchmark cannot diagnose the failure mode that matters most for self-evolution: a new tool that compiles and passes per-tool unit tests but is incompatible with previously-created tools (signature drift, namespace collisions, redundant utilities). A composition-required subtask family is missing.
2. **"Abstract requirements" is itself an unmeasured specification.** The novelty claim hinges on "without preset specifications, only abstract requirements." But abstract requirements are still natural-language specs whose abstraction *level* is not standardized. Without inter-author reliability for prompt-specs (κ across humans rewriting the same task) the FC and interface metrics conflate (a) model failure with (b) prompt-spec ambiguity. A prompt-paraphrase robustness study would isolate this.
3. **Interface oracle is a hidden preset spec.** Interface-compliance metrics presumably check the produced signature against some reference. If the reference is a hand-coded oracle interface, the paper has reintroduced exactly the "preset specification" it claims to avoid — just at a different layer. The paper should disclose how the oracle interface is derived and whether models that produce a *correct-but-different* interface are penalised.
4. **Multiple-comparison inflation.** Across L1/L2/L3/L4 × ~tasks × models, the per-cell sample sizes are small. The "even SOTA models fail" claim and the "minor flaws amplified through pipeline" claim should be backed by per-axis CIs and a Bonferroni- or Holm-corrected family-wise error treatment, not raw point comparisons.
